# Trade-In Migration Plan
## tradein.cambridgetcg.com → cambridgetcg.com/trade-in

**Source:** `cambridge-tcg-site/web/tradein/` (vanilla JS, Cloudflare Workers, static buylist.json)
**Target:** `cambridgetcg-storefront/` (Next.js 16, App Router, Tailwind, wholesaletcgdirect.com API)
**Reference doc:** TRADEIN_FLOW.md (full edge cases, policies, email templates, data model)

---

## Current State Audit

### What exists (tradein.cambridgetcg.com)
- Browse-only buylist — 486+ One Piece cards with cash/credit prices
- Hash-router SPA (`#/`, `#/cart`, `#/submit`, `#/confirm/:ref`, `#/status`, `#/terms`)
- Sell cart (localStorage) with quantity controls and running total
- Submit form (name, email, payment preference, delivery method)
- Cloudflare Worker API (`tradein-api.cambridgetcg.com`) — handles POST /tradein
- Static buylist.json generated by `build-buylist.py` from wholesale DB
- Want tiers: stock=0 → want 4, stock 1-2 → want 2, stock 3+ → want 0
- Cash: 55% of landed cost. Credit: 77% of landed cost.

### What's missing (Phase 1 gaps)
- No email confirmation to customer (SNS notification to store only)
- No status check for customers
- No admin panel for store staff
- No grading workflow
- No database (submissions not persisted beyond SNS notification)

---

## Migration Architecture

### Why migrate?
1. Unified domain (cambridgetcg.com/trade-in) — better SEO, single brand
2. Real database (wholesale RDS) — persistent submissions, status tracking
3. Email infrastructure (AWS SES) — customer communication
4. Shared auth + components with main storefront
5. Direct API access to wholesale DB for live buylist prices (no static JSON build step)

### Stack
- **Frontend:** Next.js 16 App Router, Tailwind (matches storefront)
- **API:** Next.js Route Handlers (replaces Cloudflare Worker)
- **Database:** Wholesale RDS PostgreSQL (add tradein_submissions, tradein_items, tradein_register tables)
- **Email:** AWS SES (tradein@cambridgetcg.com)
- **Buylist data:** wholesaletcgdirect.com/api/v1/prices (live, no build step)

#### Buylist Price Logic
```typescript
// Cash price: 55% of wholesale landed cost
// Credit price: 77% of wholesale landed cost
// Want tier (cash): stock=0 → 4, stock 1-2 → 2, stock 3+ → 0
// Want tier (credit): always 99 (unlimited)

function computeTradePrices(card: PriceItem) {
  const cashPrice = +(card.price_gbp * 0.55).toFixed(2);
  const creditPrice = +(card.price_gbp * 0.77).toFixed(2);
  const cashWant = card.stock === 0 ? 4 : card.stock <= 2 ? 2 : 0;
  const creditWant = 99;
  return { cashPrice, creditPrice, cashWant, creditWant };
}
```

#### Database Migrations
```sql
-- tradein_submissions
CREATE TABLE tradein_submissions (
  id SERIAL PRIMARY KEY,
  reference VARCHAR(20) UNIQUE NOT NULL,  -- TI-YYYYMMDD-XXXX
  status VARCHAR(30) NOT NULL DEFAULT 'submitted',
  customer_name VARCHAR(200) NOT NULL,
  customer_email VARCHAR(200) NOT NULL,
  customer_phone VARCHAR(30),
  payment_method VARCHAR(10) NOT NULL,    -- 'cash' | 'credit'
  bank_sort_code VARCHAR(10),
  bank_account_number VARCHAR(10),
  delivery_method VARCHAR(10) NOT NULL,   -- 'mail' | 'instore'
  is_over_18 BOOLEAN NOT NULL DEFAULT false,
  notes TEXT,
  quoted_cash_total NUMERIC(10,2),
  quoted_credit_total NUMERIC(10,2),
  final_total NUMERIC(10,2),
  tracking_number VARCHAR(100),
  payment_reference VARCHAR(100),
  quote_expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- tradein_items
CREATE TABLE tradein_items (
  id SERIAL PRIMARY KEY,
  submission_id INTEGER REFERENCES tradein_submissions(id),
  sku VARCHAR(30) NOT NULL,
  quantity INTEGER NOT NULL,
  quoted_cash_price NUMERIC(10,2),
  quoted_credit_price NUMERIC(10,2),
  accepted_qty INTEGER,
  condition_grade VARCHAR(5),             -- NM, LP, MP
  final_unit_price NUMERIC(10,2)
);

-- tradein_register (legal compliance UK)
CREATE TABLE tradein_register (
  id SERIAL PRIMARY KEY,
  submission_id INTEGER REFERENCES tradein_submissions(id),
  seller_name VARCHAR(200),
  seller_email VARCHAR(200),
  seller_id_reference VARCHAR(100),       -- required for >£500
  items_description TEXT,
  total_paid NUMERIC(10,2),
  payment_method VARCHAR(10),
  date DATE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

#### API Route: POST /api/tradein/submit
1. Validate cart items against live buylist (re-fetch prices, check <10% drift)
2. Generate reference: `TI-YYYYMMDD-{4-digit-seq}`
3. Insert tradein_submissions + tradein_items
4. Send SES confirmation email to customer
5. Send SNS/SES notification to contact@cambridgetcg.com
6. Return `{ reference, expiresAt }`

- Mark as `expired`, send expiry email to customer

**Goal:** Store staff can manage submissions without touching the DB

```
src/app/trade-in/admin/

  [ref]/route.ts              ← GET/PATCH: view + update status
  [ref]/grade/route.ts        ← POST: record grading per item
  [ref]/payment/route.ts      ← POST: issue credit code or record bank transfer
```

#### Auth
- Use next-auth (already installed) with email magic-link or simple shared secret
- Admin routes protected by middleware checking session role

#### Admin Actions Per Submission
- Mark as Received → sends "Cards Received" email
- Record grading (per item: NM/LP/MP, accepted_qty)
- Issue payment:
  - Credit: generate discount code via Shopify API (or manual entry)
  - Cash: record bank transfer reference
- Mark as Completed
- Request return (sends return instructions email)

## Key Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Buylist data source | wholesaletcgdirect.com API (live) | No build step, always current prices |
| Cart persistence | localStorage (same as main cart) | No auth required |
| Email provider | AWS SES | Already in stack, cheapest |

| Bank details storage | Encrypted in RDS | Avoid collecting separately |
| Credit payment | Shopify discount code (manual initially) | Simple Phase 1, automate later |

## ⚠️ Volume Implication — Architecture Revision

5,000–10,000/week = ~700–1,400/day = ~1 per minute at peak.

- Email: 700–1,400 confirmation emails/day → SES at ~£1.50/day
- API: rate limiting mandatory (max 5/IP/hour, max 3/email/day)
- Admin panel: critical for Phase 1, not Phase 3 — cannot manage this by email
- Dedup: fingerprint by email + cart hash within 10 min window
- Queue: async email with retry (SES is fast but add resilience)
- DB pooling: PgBouncer or Drizzle pool config from day 1
